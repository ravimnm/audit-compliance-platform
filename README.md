# Secure Multi-Tenant Audit & Compliance Platform (SMTAP)

A secure, multi-tenant audit logging and compliance platform designed for **tamper-evident audit storage, integrity verification, and centralized runtime security telemetry collection**.

SMTAP provides a trusted backend for storing, verifying, and investigating security events generated by enterprise applications and runtime security agents such as **JRSA (Java Runtime Security Agent)**.

---

## 🚀 Features

* JWT-based Authentication
* Role-Based Access Control (RBAC)
* Multi-Tenant Isolation
* Tamper-Evident Audit Logs using Hash Chaining
* Runtime Security Event Ingestion
* Centralized Security Telemetry Collection
* Integrity Verification APIs
* Dynamic Filtering, Pagination, and Sorting
* CSV Export
* Swagger/OpenAPI Documentation
* External Integrity Anchoring

## 🧠 Core Integrity Model

Each audit event stores:

* `hash`
* `previousHash`

Hash computation:

```text
H(n) = SHA256(H(n-1) + EventData)
```

EventData includes:

* actorId
* role
* tenantId
* action
* timestamp

---

## 🔗 Chain Behavior

```text
Log1 → Log2 → Log3 → Log4 → ...
```

If any historical log is modified:

* Chain verification fails
* Subsequent hashes become invalid
* Tampering is detected deterministically

---

## 🔐 Threat Model

### System Detects

* Application-level log modification
* Accidental corruption
* Single-record database tampering
* Runtime security events from protected applications
* Suspicious JVM activities reported by JRSA

### System Does NOT Protect Against

* Full database rewrite with recomputation
* Compromised application server
* Secret key compromise
* External integrity anchor deletion
* Insider attacks with unrestricted infrastructure access

## ⚡ Runtime Security Integration

SMTAP acts as the centralized backend for **JRSA**.

JRSA forwards runtime security events such as:

* Process Execution
* Reflection Usage
* Suspicious Command Execution
* Runtime Policy Violations

Example event:

```json
{
  "eventType": "PROCESS_EXECUTION",
  "severity": "HIGH",
  "source": "JRSA",
  "details": "powershell.exe -enc SQBtACAAbQBhAGwAaQBjAGkAbwB1AHMA"
}
```

These events are:

* Persisted in PostgreSQL
* Linked using tamper-evident hash chains
* Available for investigation and export

## 📌 APIs

### Authentication

Generate token:

```http
GET /auth/token?userId=user1&role=ADMIN&tenantId=tenant_1
```

---

### Create Audit Event

```http
POST /audit/events
```

Body:

```json
{
  "action": "USER_LOGIN"
}
```

---

### Ingest Runtime Security Event

```http
POST /security/events
```

Body:

```json
{
  "eventType":"PROCESS_EXECUTION",
  "severity":"HIGH",
  "source":"JRSA",
  "details":"cmd.exe execution detected"
}
```

---

### Retrieve Audit Logs

```http
GET /audit/events
```

Supports:

* Filtering
* Pagination
* Sorting
* Date Ranges

---

### Verify Integrity

```http
GET /audit/events/verify
```

---

### Export Logs

```http
GET /audit/events/export
```

## ⚙️ Configuration

Configure `application.properties`:

```properties
spring.datasource.url=jdbc:postgresql://localhost:5432/audit_platform
spring.datasource.username=postgres
spring.datasource.password=yourpassword

jwt.secret=your-secret-key
```

## 🧪 Tampering Simulation

1. Insert logs using APIs

2. Modify database manually:

```sql
UPDATE audit_logs
SET action='HACKED'
WHERE id=1;
```

3. Verify integrity:

```http
GET /audit/events/verify
```

Expected:

```text
Tampering detected at log ID: 1
```

---

## ⚠️ Limitations

* Detects tampering but does not prevent it
* Full chain recomputation remains possible
* No digital signatures
* No append-only database enforcement
